Generate a multi-tenant API rate limiting and quota enforcement analysis system. Read all CSV input files from the task directory to process API request logs and compute quota consumption, violations, throttling decisions, and billing for each tenant.

For request_attribution.csv, process each API request chronologically to track running consumption. For each request, determine effective_tokens by using the actual tokens_used if present, otherwise estimate using the endpoint avg_token_estimate from the endpoints table, but if response_code is 500 or higher then effective_tokens is zero. Compute effective_cost by multiplying effective_tokens by the endpoint cost_multiplier and rounding to 2 decimals with 4 decimal intermediate precision. Track hourly_count as the running count of requests per tenant per clock hour where hours reset at the top of each hour. Track daily_count and daily_tokens as running totals per tenant per calendar day. Determine within_limit by comparing against the tenant tier limits where YES means all limits satisfied, BURST means hourly exceeded but within burst multiplier, and NO means any limit exceeded. Set throttle_reason to the pipe-separated list of exceeded limits like DAILY_EXCEEDED or HOURLY_EXCEEDED or TOKEN_EXCEEDED. Set billable to NO if response_code is 400-499 or 500+ or endpoint billable is NO, otherwise YES. Apply any custom_rate_limit from the tenant record to override hourly limits. Apply quota_overrides that are active on the request timestamp. Output request_id, tenant_id, endpoint_id, timestamp, effective_tokens, cost_multiplier, effective_cost, quota_bucket as the date string, hourly_count, daily_count, daily_tokens, within_limit, throttle_reason, and billable.

For tenant_consumption.csv, aggregate request attribution by tenant and quota_bucket which is the calendar date. Sum request_count and tokens_consumed and total_cost. Look up the tenant tier to get request_limit and token_limit. Compute request_pct as request_count divided by request_limit times 100 rounded to 2 decimals. Compute token_pct similarly. Set quota_used_pct to the max of request_pct and token_pct rounded to 2 decimals. Determine status where THROTTLED means violation_count exceeds 10, OVERAGE means quota_used_pct exceeds 100, WARNING means quota_used_pct exceeds 80, otherwise NORMAL. Compute overage_requests as max of zero and request_count minus request_limit. Compute overage_tokens similarly. Output tenant_id, tier_id, quota_bucket, request_count, tokens_consumed, total_cost, request_limit, token_limit, request_pct, token_pct, quota_used_pct, status, violation_count as the count of requests where within_limit was NO, overage_requests, and overage_tokens. Sort by tenant_id then quota_bucket.

For violation_ledger.csv, create one record for each request where within_limit was NO. Track violations per tenant including historical_violations that are resolved. For each violation, compute violation_number as the count of prior violations for that tenant plus one including historical. Match against penalty_rules by checking if violation_number falls between min_violations_in_period and max_violations_in_period. Rules with applies_to_tier matching the tenant tier or ALL apply. If multiple rules match, use the one with highest flat_penalty_usd. Extract severity, flat_penalty, pct_penalty, and enforcement_action from the matched rule. Set penalty_amount to flat_penalty rounded to 2 decimals. Compute cumulative_penalty as the running sum of penalty_amount for that tenant. Output violation_id in VN000001 format, tenant_id, request_id, violation_timestamp, violation_type as the first segment of throttle_reason before any pipe, violation_number, severity, flat_penalty, pct_penalty, penalty_amount, cumulative_penalty, and enforcement_action. Sort by tenant_id then violation_timestamp.

For throttle_decisions.csv, output one record for each request where within_limit was NO. For each, determine the decision based on the tenant tier throttle_behavior and grace_period_minutes. If the tenant exempt_from_throttling is YES then decision is ALLOW with decision_reason EXEMPT_TENANT. Otherwise track the first violation timestamp per tenant. If the current request is within grace_period_minutes of the first violation and grace has not been used then decision is GRACE with decision_reason indicating the grace period. Once past grace, if throttle_behavior is HARD_BLOCK then decision is BLOCK with reason HARD_BLOCK_TIER. For SOFT_THROTTLE, compute hourly_excess as hourly_count minus 100. If hourly_excess exceeds 50 then BLOCK with SEVERE_OVERAGE. If hourly_excess exceeds 20 then THROTTLE_90 with HIGH_OVERAGE. Otherwise THROTTLE_50 with MODERATE_OVERAGE. Output request_id, tenant_id, tier_id, timestamp, throttle_reason, hourly_count, daily_count, decision, and decision_reason. Sort by tenant_id then timestamp.

For billing_summary.csv, aggregate across all quota_buckets per tenant. Compute base_cost by multiplying the tier base_monthly_price by the fraction of the month covered which is 3 divided by 30, rounded to 2 decimals. Compute overage_cost by multiplying overage_requests divided by 1000 by overage_rate_per_1k_requests plus overage_tokens divided by 1000 by overage_rate_per_1k_tokens, both rounded to 2 decimals then summed. Sum penalty_amount from violation_ledger to get penalty_cost rounded to 2 decimals. Compute total_due as base_cost plus overage_cost plus penalty_cost rounded to 2 decimals. Count violations from violation_ledger. Output tenant_id, tier_id, billing_period as the date range string, total_requests, total_tokens, overage_requests, overage_tokens, base_cost, overage_cost, penalty_cost, total_due, and violation_count. Sort by tenant_id.
