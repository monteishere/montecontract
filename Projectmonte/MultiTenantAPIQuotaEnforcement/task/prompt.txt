You receive data from a multi-tenant API platform covering requests made by 20 tenants across a three-day billing period: 2025-01-01 through 2025-01-03. Ten input files describe the tenants, their tier policies, endpoints, request logs, quota overrides, surge pricing windows, endpoint weight multipliers, SLA capacity schedules, penalty rules, and each tenant's pre-existing violation history. Your task is to produce five output files: one attributing resource consumption to each request, one summarising daily consumption per tenant, one recording every quota violation with its severity and penalty, one capturing throttle enforcement decisions, and one computing the final invoice per tenant for the period.

api_requests.csv has columns request_id, tenant_id, endpoint_id, timestamp, tokens_used, response_code.
tenants.csv has columns tenant_id, tier_id, custom_rate_limit (when non-null overrides the tier's hourly_request_limit for that tenant), exempt_from_throttling.
tier_policies.csv has columns tier_id, hourly_request_limit, daily_request_limit, daily_token_limit, burst_multiplier, grace_period_minutes, throttle_behavior, base_monthly_price, overage_rate_per_1k_requests, overage_rate_per_1k_tokens.
endpoints.csv has columns endpoint_id, category, avg_token_estimate, cost_per_token, billable.
surge_windows.csv has columns surge_window_id, category, effective_start_utc, effective_end_utc, multiplier. A surge window applies to a request when the request timestamp satisfies effective_start_utc <= timestamp < effective_end_utc and the window's category matches the endpoint category.
endpoint_weights.json is a nested structure mapping endpoint category to tenant tier to a weight multiplier. Missing category-tier combinations have a weight of 1.0.
sla_schedule.xlsx contains SLA capacity multipliers. Use only the sheet named active_policies; the other sheet contains legacy data and must be ignored. The active_policies sheet has columns tier_id, day_of_week (MON through SUN or ANY as a catch-all), capacity_multiplier.
penalty_rules.csv has columns rule_id, applies_to_tier, min_prior_violations, max_prior_violations, severity, flat_penalty_usd, pct_penalty_per_1k_tokens, cap_multiplier, enforcement_action.
quota_overrides.csv has columns tenant_id, override_field, override_value, valid_from, valid_to. Only rows whose override_field exactly equals one of hourly_request_limit, daily_request_limit, or daily_token_limit are applied; all others are silently ignored.
historical_violations.csv has columns tenant_id, violation_count representing violations accumulated before this billing period.

Process requests in ascending timestamp order, using request_id as the tiebreaker. A request's base_tokens is zero when response_code is 5xx, avg_token_estimate when tokens_used is null, and tokens_used otherwise. The effective_tokens for a request equals base_tokens multiplied by a weight_multiplier from endpoint_weights.json (looked up by endpoint category and tenant tier) and a surge_multiplier from the matching surge window if any (1.0 if none applies). The effective hourly request limit for a request is floor(capacity_multiplier × base_hourly_limit), where capacity_multiplier comes from the active_policies SLA sheet by matching tier and the day-of-week of the request timestamp, falling back to the ANY row, then to 1.0. Any applicable quota_override that is active at the request timestamp replaces the corresponding limit before this floor is applied.

A request's within_limit is YES when all running counts are within their limits; BURST when the hourly request count exceeds the hourly limit but remains within burst_multiplier times that limit; NO otherwise. The running hourly_request_count and daily_request_count reset at the top of each hour and the start of each calendar day respectively. The daily_token_count is the running sum of effective_tokens within the calendar day.

For violation numbering, a tenant's prior_violation_count for their current violation equals the tenant's historical violation count from historical_violations.csv plus the count of new violations already recorded for that tenant before the current one. The violation_number equals prior_violation_count plus one. The matching penalty rule has applies_to_tier equal to the tenant's tier or ALL, and prior_violation_count falls within the rule's min_prior_violations to max_prior_violations range inclusive. When multiple rules match, use the one with the highest flat_penalty_usd. The penalty_amount equals flat_penalty_usd plus the lesser of (pct_penalty_per_1k_tokens times effective_tokens divided by 1000) and (cap_multiplier times flat_penalty_usd), rounded to 2 decimal places.

A throttle decision is generated for every request where within_limit is NO. If the tenant's exempt_from_throttling is YES, the decision is ALLOW with reason EXEMPT_TENANT. Otherwise, if the elapsed minutes since the tenant's first violation in the current session is within the tier's grace_period_minutes and grace has not yet been consumed for this tenant, the decision is GRACE with reason WITHIN_GRACE_Xmin where X is the tier's grace_period_minutes. After grace is exhausted, HARD_BLOCK tiers produce BLOCK with reason HARD_BLOCK_TIER. SOFT_THROTTLE tiers compute hourly_excess as hourly_request_count minus effective_hourly_limit: excess above 50 gives BLOCK/SEVERE_OVERAGE, above 20 gives THROTTLE_90/HIGH_OVERAGE, otherwise THROTTLE_50/MODERATE_OVERAGE.

Billing covers the period 2025-01-01 to 2025-01-03. The base_cost is the tier's base_monthly_price prorated as (count of distinct calendar days in the tenant's data divided by the number of calendar days in the billing month), rounded to 2 decimal places. The billing_period string is "2025-01-01 to 2025-01-03".

Produce the five output files below. Write them to the task/ directory. Do not write output to any other location.

request_attribution.csv — one row per request, sorted by timestamp then request_id.
Columns: request_id, tenant_id, endpoint_id, timestamp, base_tokens, weight_multiplier, surge_multiplier, effective_tokens, hourly_request_count, daily_request_count, daily_token_count, effective_hourly_limit, within_limit, throttle_reason, billable.
billable is NO if response_code is 4xx or 5xx or the endpoint's billable flag is NO; otherwise YES. throttle_reason is a pipe-separated list of HOURLY_EXCEEDED, DAILY_EXCEEDED, TOKEN_EXCEEDED for each exceeded limit; empty string when within_limit is YES or BURST.

tenant_consumption.csv — one row per (tenant_id, calendar date), sorted by tenant_id then quota_bucket.
Columns: tenant_id, tier_id, quota_bucket, request_count, tokens_consumed, total_cost, request_limit, token_limit, request_pct, token_pct, quota_used_pct, status, violation_count, overage_requests, overage_tokens.
quota_bucket is the date string YYYY-MM-DD. request_pct = request_count / request_limit × 100 (2 dp). token_pct = tokens_consumed / token_limit × 100 (2 dp). quota_used_pct = max(request_pct, token_pct). status: THROTTLED if violation_count > 10; OVERAGE if quota_used_pct > 100; WARNING if quota_used_pct > 80; NORMAL otherwise. violation_count = count of requests with within_limit = NO for that tenant on that date. total_cost = sum of effective_tokens × cost_per_token for billable requests. overage_requests = max(0, request_count − request_limit). overage_tokens = max(0, tokens_consumed − token_limit).

violation_ledger.csv — one row per request where within_limit = NO, sorted by tenant_id then violation_timestamp.
Columns: violation_id, tenant_id, request_id, violation_timestamp, violation_type, prior_violation_count, violation_number, severity, flat_penalty_usd, pct_penalty_per_1k_tokens, penalty_amount, cumulative_penalty, enforcement_action.
violation_id is VN followed by a 6-digit zero-padded integer starting at VN000001. violation_type is the first pipe-segment of throttle_reason. cumulative_penalty is the running sum of penalty_amount for that tenant (2 dp).

throttle_decisions.csv — one row per request where within_limit = NO, sorted by tenant_id then timestamp.
Columns: request_id, tenant_id, tier_id, timestamp, throttle_reason, hourly_request_count, daily_request_count, effective_hourly_limit, decision, decision_reason.

billing_summary.csv — one row per tenant, sorted by tenant_id.
Columns: tenant_id, tier_id, billing_period, total_requests, total_tokens, overage_requests, overage_tokens, base_cost, overage_cost, penalty_cost, total_due, violation_count.
overage_cost = (overage_requests / 1000 × overage_rate_per_1k_requests) + (overage_tokens / 1000 × overage_rate_per_1k_tokens), rounded to 2 dp. penalty_cost = sum of penalty_amount from violation_ledger for this tenant (2 dp). total_due = base_cost + overage_cost + penalty_cost (2 dp).
